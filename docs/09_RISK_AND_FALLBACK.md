# 09 — Risk Register and Fallback Plan

## 1. External API failure

### Risk

Search or LLM provider is unavailable, slow or rate-limited.

### Prevention

- Test keys before the event.
- Add explicit timeouts.
- Limit queries.
- Limit search results.
- Use a smaller/faster configured model for intermediate agents.
- Cache demo input results during rehearsal when allowed.

### Fallback

- Set `MOCK_MODE=true`.
- Show a visible Demo Data badge.
- Never present fixture evidence as live search.

## 2. LLM returns malformed JSON

### Prevention

- Structured output.
- Small schemas.
- Clear enums.
- Low temperature.
- Separate agents.

### Fallback

- One repair retry.
- Typed default with error reason.
- Continue partial report.

## 3. Hallucinated sources

### Prevention

- Evidence IDs generated by backend.
- Experts receive only normalized evidence objects.
- Judge validates every citation.
- Report generator receives approved findings only.

### Fallback

- Remove invalid citation.
- Add limitation.
- Lower confidence.

## 4. Latency

### Prevention

- Two-endpoint design.
- Run experts concurrently.
- Three search queries maximum.
- Five to eight evidence items maximum.
- One retry maximum.
- No database.

### Fallback

- Show Fast Warning.
- Cancel or skip failed expert after timeout.
- Return `status=partial`.
- Use available evidence.

## 5. Frontend/backend integration mismatch

### Prevention

- JSON schemas committed first.
- Generate or manually mirror TypeScript types.
- Keep example responses.
- Contract tests.

### Fallback

- Render top-level report summary.
- Add safe optional chaining.
- Keep raw JSON debug panel behind a development flag.

## 6. Deployment failure

### Prevention

- Local run verified.
- `.env.example`.
- CORS configured.
- Health endpoint.
- Build commands documented.

### Fallback

- Run both services locally.
- Use a local tunnel only when already tested.
- Present backup video if network is unavailable.

## 7. Merge conflicts

### Prevention

- One owner per file area.
- Small branches.
- Frequent integration.
- No broad formatting.

### Fallback

- Keep `main` untouched.
- Cherry-pick task commits.
- Drop nonessential polish.

## 8. Score appears overconfident

### Prevention

- Separate risk and confidence.
- Confidence penalties.
- Clear wording.
- Limitation section.

### Fallback

- Rename “probability” to “risk score”.
- Remove exact percentage from pitch if calibration is not validated.
- Use risk bands.

## 9. Sensitive data

### Prevention

- No persistence.
- No analytics containing raw text.
- No secret logging.
- Privacy notice.

### Fallback

- Clear logs.
- Use synthetic demo data.
